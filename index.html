Terminologies 1/2

  • core ≈ une table en relationnel
  • schema ≈ DESCRIBE TABLE "MaTable";
  • field ≈ une colone en relationnel
  • document ≈ un n-uplet (une ligne) en relationnel
[schema multi core]

Terminologies 2/2

  • n core
    • 1 schema : schema.xml
      • n fieldType : définitions des type des données disponibles
        • 0 à 2 analyzer : un à l'indexation et un à la requête
          • 1 tokenizer : découpeur de token
          • n filter : modifieurs de token
      • n field
        • name : nom du champs
        • type : type de données du champs (cf. fieldType)

        • indexed / stored / multivalued / required
__analyzer__ : "*Analyzers are components that pre-process input text at index time and/or at search time.*" __tokenizer__ : Unique "*A Tokenizer splits a stream of characters (from each individual field value) into a series of tokens.*" __filter__ : "*Tokens produced by the Tokenizer are passed through a series of Token Filters that add, change, or remove tokens.*"

Solr vs SGBD relationnels

  • Pas de jointure complexe (un système de jointure simple existe depuis Solr 4.0)
  • Pas de sous-requêtes (enfin si, mais inutilisable)
  • Transactionnel aux coeurs du logiciel : toutes modification de donnée ce fait dans une transaction (commit/rollback obligatoire)
  • "egal"

Solr dans le SI

Un moteur de recherche vient en supplément de BDD existantes (relationnelle, NoSQL, fichier...)

Il ne peut pas et ne doit pas remplacer une BDD "long terme"

Indexation

L'indexation consiste à stocker des données au sein de Solr

C'est durant cette phase que Solr construit ses index internes qui lui pemettent d'être très performant

Pour indexer des données il suffit de faire une requête HTTP le requestHandler d'update du core

Les données doivent être envoyé en XML ou JSON

Le processus d'indexation

Il existe trois méthodes pour gérer sont indexation :


"Full indexation"" : un bach d'indexation ré-indexe toutes les données à interval régulier

"Indexation incrémentale" : un bach d'indexation indexe les nouvelles données et les données modifiés à interval régulier

"Indexation en live" : lors de la mise à jours des données dans les applications, mettre à jours les données dans Solr


Aucune n'est parfaite. Il faut choisir en fonction de son application

Plusieurs de ces techniques peuvent/doivent être utiliser pour en paralelle :
il est toujours utile de pouvoir ré-indexer tout un core à partir de zéro
(oui Solr plante des fois)

Paramètres de base

  • q : coeurs de la requête de recherche
  • fq : clause "where" de filtre des résultat (ne perturbe pas la clause q)
  • fl : liste des champs du document a retourner
  • rows : nombre de document maximum
  • start : offset
  • sort : ordonancement des résulats
